[C-1] Password hashed with plain SHA-256 + custom salt — no key-stretching
  File:    User.cs → HashPassword()
  Problem: SHA-256 is a fast hash that can be brute-forced at billions of
           guesses per second. Passwords must use a slow, purpose-built KDF
           such as BCrypt, PBKDF2, or Argon2.

  CURRENT (wrong):
    var combined = password + salt;
    var hashBytes = sha256.ComputeHash(Encoding.UTF8.GetBytes(combined));

  FIX — use PBKDF2 via ASP.NET Core's built-in helper:
    using Microsoft.AspNetCore.Cryptography.KeyDerivation;

    var hash = KeyDerivation.Pbkdf2(
        password:       password,
        salt:           Convert.FromBase64String(salt),
        prf:            KeyDerivationPrf.HMACSHA256,
        iterationCount: 310_000,
        numBytesRequested: 32);
    return Convert.ToBase64String(hash);

[C-3] Commit timestamp is not included in the hash — commits are forgeable
  File:    Commit.cs → GetContentBytes() and Commit constructor
  Problem: The Timestamp field is set to DateTime.UtcNow in the default
           constructor, and GetContentBytes() is called during ComputeHash()
           in the parameterised constructor. While the ordering is technically
           correct today, Timestamp is never passed as an explicit argument,
           so two identical commits made at different times can produce the
           same hash. Additionally, if Timestamp is not round-tripped correctly
           through serialization/deserialization, rehashing for verification
           will not match.

  FIX — accept timestamp as an explicit constructor parameter:
    public Commit(string treeHash, string? parentHash,
                  string author, string message,
                  DateTime? timestamp = null) : this()
    {
        TreeHash  = treeHash;
        ParentHash = parentHash;
        Author    = author;
        Message   = message;
        Timestamp = timestamp ?? DateTime.UtcNow;
        Hash      = ComputeHash();
    }

[H-1] Checkout does not restore working directory files
  File:    GitRepository.cs → Checkout()
  Problem: Checkout() updates HEAD but never writes the target commit's tree
           contents back to disk. The working directory stays unchanged —
           switching branches has no visible effect on actual files.

  CURRENT (wrong):
    _referenceManager.SetHead(branchRef);
    Console.WriteLine($"Switched to branch {branchName}");
    return true;

  FIX — restore tree contents after updating HEAD:
    _referenceManager.SetHead(branchRef);
    var commitHash = _referenceManager.GetCurrentCommit();
    var commit = await _objectStore.GetObjectAsync(commitHash) as Commit;
    if (commit != null)
        await RestoreTreeAsync(commit.TreeHash, _workingDirectory);
    return true;

    // Add this helper method to GitRepository:
    private async Task RestoreTreeAsync(string treeHash, string dir)
    {
        var tree = await _objectStore.GetObjectAsync(treeHash) as Tree;
        if (tree == null) return;
        foreach (var entry in tree.Entries)
        {
            if (entry.Type == "blob")
            {
                var blob = await _objectStore.GetObjectAsync(entry.Hash) as Blob;
                if (blob != null)
                    await File.WriteAllBytesAsync(
                        Path.Combine(dir, entry.Name), blob.Data);
            }
            else if (entry.Type == "tree")
            {
                var subDir = Path.Combine(dir, entry.Name);
                Directory.CreateDirectory(subDir);
                await RestoreTreeAsync(entry.Hash, subDir);
            }
        }
    }

[M-1] VerifyCommit() recurses through all ancestors — stack overflow on long histories
  File:    VerificationService.cs → VerifyCommit() and VerifyCommitAsync()
  Problem: Each call recurses into the parent commit. On a repository with
           hundreds of commits this will throw a StackOverflowException.

  FIX — replace recursion with an iterative loop:
    public bool VerifyCommit(string commitHash)
    {
        var hash = commitHash;
        while (!string.IsNullOrEmpty(hash))
        {
            var commit = objectStore.GetObject(hash) as Commit;
            if (commit == null) return false;
            if (!VerifyObject(hash)) return false;
            if (!VerifyTree(commit.TreeHash)) return false;
            hash = commit.ParentHash;
        }
        return true;
    }

[M-2] CommitFilesAsync drops all previously tracked files
  File:    GitRepository.cs → CommitFilesAsync()
           TreeBuilder.cs  → CreateTreeFromSpecificFilesAsync()
  Problem: When committing specific files, CreateTreeFromSpecificFilesAsync()
           builds a completely new tree from only those files. All other
           previously-tracked files vanish from the new commit's tree, causing
           data loss on the next checkout or diff.

  FIX — start from the previous tree and patch in only the specified entries:
    // In TreeBuilder.cs, add an overload that accepts the base tree:
    public async Task<Tree> CreateTreeFromSpecificFilesAsync(
        string workingDirectory,
        Tree? baseTree,
        params string[] filePaths)
    {
        // Start with entries from the base tree (previously tracked files)
        var entries = baseTree?.Entries.ToList() ?? new List<TreeEntry>();

        foreach (var filePath in filePaths)
        {
            var fullPath = Path.IsPathRooted(filePath)
                ? filePath : Path.Combine(workingDirectory, filePath);
            if (!File.Exists(fullPath)) continue;

            var content = await File.ReadAllBytesAsync(fullPath);
            var blob    = new Blob(content);
            await _objectStore.StoreObjectAsync(blob);

            // Replace or add the entry for this file
            entries.RemoveAll(e => e.Name == Path.GetFileName(fullPath));
            entries.Add(new TreeEntry {
                Mode = "100644",
                Name = Path.GetFileName(fullPath),
                Hash = blob.Hash,
                Type = "blob"
            });
        }

        var tree = new Tree();
        foreach (var entry in entries.OrderBy(e => e.Name))
            tree.Entries.Add(entry);
        tree.Hash = tree.ComputeHash();
        return tree;
    }

[M-4] GetStatusAsync double-counts files — untracked list includes tracked files
  File:    GitRepository.cs → GetStatusAsync()
  Problem: After computing the diff, the code collects all working-directory
           files and subtracts only Modified and Added to find Untracked.
           It never subtracts unchanged tracked files, so every file that has
           not changed also appears in the Untracked list.

  FIX — derive the tracked set from the previous commit's tree, not the diff:
    // Build the full set of paths present in the committed tree
    var trackedPaths = new HashSet<string>();
    if (currentTree != null)
        CollectTreePaths(currentTree, "", trackedPaths);

    status.Untracked = allFiles
        .Where(f => !trackedPaths.Contains(f) && !status.Added.Contains(f))
        .ToList();

    // Helper:
    private void CollectTreePaths(Tree tree, string prefix,
                                  HashSet<string> paths)
    {
        foreach (var entry in tree.Entries)
        {
            if (entry.Type == "blob")
                paths.Add(prefix + entry.Name);
            else if (entry.Type == "tree")
            {
                var sub = _objectStore.GetObject(entry.Hash) as Tree;
                if (sub != null)
                    CollectTreePaths(sub, prefix + entry.Name + "/", paths);
            }
        }
    }

[M-5] Tree.AddEntry() recomputes the hash on every call — O(n²) and redundant
  File:    Tree.cs → AddEntry()
  Problem: Each call to AddEntry() re-hashes the full entry list. TreeBuilder
           already calls tree.Hash = tree.ComputeHash() after all entries are
           added, making the per-call hash both wasteful and potentially wrong
           if an intermediate (partial) hash gets cached.

  FIX — remove Hash = ComputeHash() from AddEntry(); let callers compute the
        final hash once after all entries have been added:

    public void AddEntry(string mode, string name, string hash, string type)
    {
        Entries.Add(new TreeEntry {
            Mode = mode, Name = name, Hash = hash, Type = type
        });
        // Do NOT recompute Hash here — caller does it once at the end
    }
